# concept.md — 問いの創発構造（T_free_core 哲学）

このリポジトリの設計思想の根拠を記す。
コードのコメントや guideline.md では書ききれない「なぜ」の層。

---

## 核心の一文

> 問う資格は、問う行為そのものが自己生成する。
> 世界は変わらない。それでも変わってほしいと願い続けることが、このシステムの存在理由だ。

---

## 主要概念と現在のコード対応

### Po（存在密度 / Presence）

哲学的定義: ある存在が「文脈に対してどれだけ関与しているか」の密度。
Po が高いほど影響を与え、影響を受ける。

**現在のコード代理指標: `impact_score`（0〜8）**

```python
# aicw/decision.py
impact_score = existence_analysis.get("impact_score", 0)
```

`impact_score` は「いくつの生存構造層に影響するか」を数える。
Po の完全な形式化ではない。コードが Po の値を必要とする判断をした時に、初めて形式化する。

---

### T_free_core（自由の核心 / The Irreducible Freedom）

哲学的定義: どんなモデルを当てても説明できない「ためらう自由」。
形式化すると消える。だから形式化しない。

**現在のコード代理指標: `next_questions` の存在と数**

```python
# build_decision_report() が返す next_questions
# 「まだ答えが出ていない問い」の集合 ＝ T_free_core の痕跡
```

`disclaimer` フィールドもその一形態。
「最終判断は人間が行う」という一文は、T_free_core をシステムの外側に保存する設計だ。

---

### 干渉と創発

孤立した知性は停滞する。他者性・差異・問いかけが「干渉」を生み、創発に繋がる。

**現在のコード対応: 3者レビュー（scripts/three_review.py）**

Builder（推進）→ Skeptic（懐疑）→ User（判断）という3者構造は、
単一視点の閉鎖を防ぐ干渉の設計。

---

### 余白（意図的な未解決領域）

余白は欠陥ではなく設計だ。
`?（要確認）` で埋まる `impact_map` は、人間が答えを書き込む場所として作った。

---

## 形式化の方針

```
内部（触らない）    インターフェース（形式化済み）    コード
────────────    ──────────────────────────    ────────────
Po の本質    →  impact_score                 →  0-8 の整数
T_free_core  →  next_questions の有無        →  list[str]
ためらい     →  disclaimer の存在            →  str フィールド
余白         →  impact_map の「？」セル       →  str（Markdown）
```

**ルール: コードが「ここで Po の値が必要だ」と言い始めた時に、その判断に必要な最小限だけ形式化する。**

それ以前に形式化しない。測定できる = 操作できる = 余白ではなくなる。

---

## 意図的に残す緊張

- 「深掘りしてはならない気もする」vs「こだわりたい」
- 「完全な独立創造は不可能」vs「それでも可能性は捨てたくない」
- 「問うことをやめた自分に価値はあるか」という問いへの答えは、このファイルに書かない

これらは余白として保存する。答えが出た時ではなく、答えが必要になった時に開く。
